### The Core Belief

Traditional product management: *Research → Write specs → Build → Test*

**AI-native product management: *Research → Build → Test → Extract specs — grounded in what actually works***

That inversion — building first, specifying second — is what I teach, practice, and help teams adopt.

### [AI Vibe Coding — Product School](https://github.com/DJN-KRS-2709/AI-Vibe-Coding---Product-School)

A complete 6-module certification program teaching product managers how to prototype with AI tools and ship real products.

**The arc:**

| Module | Theme | Students Learn To... |
|--------|-------|---------------------|
| M1 — Speed | First Prototype | Go from blank canvas to working prototype in 15 minutes |
| M2 — Aim | The Validation Lens | Design for hypotheses, not features |
| M3 — Precision | Prompt Chains | Use targeted prompt sequences instead of one-shot prompts |
| M4 — Graduation | Living PRD + Infrastructure | Extract specs from prototypes, connect GitHub + Supabase |
| M5 — Production | Backend + Database | Build real database schemas, auth, and API integrations |
| M6 — Presentation | Portfolio + Pitch | Present a complete product brief to stakeholders |

**What's in the repo:** Gamma prompts for slide generation, interactive HTML tools (Living PRD Extractor), instructor speaker notes, and the full pedagogical framework.

**Key concept — Living PRD:** Instead of writing a spec to describe something that doesn't exist, students build a working prototype first, then extract a grounded specification from it. The PRD is alive because it's rooted in a real, testable artifact.

### [PM-OS Skills Library](./skills/)

A library of **35 AI Skills** built for PM-OS (Product Management Operating System), a structured workspace where AI copilots help product managers think, write, communicate, and ship.

Each skill is a markdown instruction set that gives an AI agent specialized product management capabilities. Skills enforce rigorous PM methodology: problem-before-solution thinking, tradeoff architecture, evidence-grounded decisions, and narrative coherence.

**Highlights:**

| Category | Skills |
|----------|--------|
| Brief Pipeline | `product-brainstorm` > `write-brief` > `execute-brief` > `verify-brief` |
| Quality | `sense-check` (Ralph Wiggum Mode), `fti-groove-validator` |
| Career | `product-coach` (gap-to-next-level coaching) |
| Stakeholders | `personas` (encrypted stakeholder simulation) |
| Data | `metrics`, `vedder` (natural language to SQL) |
| Prototyping | `rapidly` (CPM + Figma + HTML from discovery docs) |
| Meta | `write-plugin` (TDD-for-skills methodology) |

**Key concept: TDD-for-Skills.** Every skill exists because an AI agent failed without it. The failure is documented, the rationalization is predicted, and the skill prevents both. Skills are behavioral contracts, not prompts.

### [Rhythm](./rhythm/)

A **34-skill integrated system** for engineering squad operations, built using a recording studio metaphor. Where PM-OS gives individual PMs superpowers, Rhythm gives entire squads an encoded operating rhythm: sprint ceremonies, delivery tracking, SDLC compliance, launch readiness, and retrospectives, all orchestrated by AI.

**The architecture:**

| Layer | What It Contains | Analogy |
|-------|-----------------|---------|
| `skills/` | 25 universal instruments | The instruments, same for every band |
| `plugins/rehearsal-room/` | 9 methodology instruments | The rehearsal space, where skills get better |
| `sheet-music/` | SDLC rules, templates, gate definitions | The genre's theory book |

**Highlights:**

| Flow | Skills Orchestrated |
|------|-------------------|
| Sprint Planning | `whos-available` > `scan-horizon` > `set-goals` > `check-health` > `forecast` |
| Sprint Close | outcome review > carry-over > health check > epic progress > status updates > velocity |
| Initiative Lifecycle | `start-discovery` > `gate-1-review` > `start-design` > `start-build` > `check-launch` > `ship-it` |
| Self-Improvement | `improve-skill` (rehearse against real data) > `save-work` (commit + ship learnings) |

**Key concept: The Rehearsal Room.** Skills improve through structured rehearsal cycles. Corrections from real usage are captured, classified (principled vs friction), and encoded as permanent lessons. The system gets smarter with every session.

**Key concept: Two-Persona Design.** Every skill must work for both "Expert Claude" (with full conversation context) and "Blank-Slate Claude" (reading only the persisted files). If knowledge only exists in conversation history, it's lost next session. The files ARE the bridge.

## How I Work With AI

I don't treat AI as a feature to bolt on. It's the operating system for how I build, teach, and lead.

**Prototyping stack:**

| Tool | Role |
|------|------|
| [Lovable](https://lovable.dev) | AI-assisted UI prototyping — from prompt to working screens |
| [Cursor](https://cursor.com) | AI-native code editor for building and refactoring at speed |
| [Supabase](https://supabase.com) | Backend infrastructure — database, auth, APIs |
| [Gamma](https://gamma.app) | AI-powered presentation and content generation |
| GitHub | Version control, collaboration, and deployment |

**Concepts I teach and practice:**

- **Prompt Chains** — Sequences of targeted prompts (Expand → Behavior → Refine) for precise prototype development
- **Living Prompt Packs** — Reusable, evolving prompt collections that encode product context and design systems
- **AI Skills** — Importable markdown instruction sets that give AI tools specialized capabilities
- **Graduation Judgment** — The discipline of knowing when a prototype is ready for infrastructure, not just when it looks good
