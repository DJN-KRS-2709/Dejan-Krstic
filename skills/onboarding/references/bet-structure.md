# Bet Structure Reference

Every active bet lives in `domains/[domain]/01_active_bets/[bet-name]/` and contains six standard files.

---

## The Six Files

### `hypothesis.md` — What we believe and how to disprove it

The core document. Contains:
- The bet statement ("We believe that X will result in Y for users Z")
- Success metrics (what "winning" looks like, with numbers)
- Falsification criteria (what would cause us to kill this bet)
- Confidence level and key assumptions

**Length:** 1 page max. If it's longer, the hypothesis is unclear.

---

### `problem_frame.md` — Why this matters

Articulates the problem before jumping to solutions. Contains:
- Current state (what's broken or missing today)
- Desired state (what success looks like for users)
- Target users (specific, not generic)
- Why now (what changed that makes this urgent)
- What we are NOT solving (explicit scope boundaries)

**Length:** 1–2 pages max.

---

### `evidence.md` — Data, research, and signals

Everything backing the hypothesis. Contains:
- Quantitative signals (metrics, usage data, experiment results)
- Qualitative signals (user research, interviews, support tickets)
- What we still need to learn
- Confidence ratings per signal

**Format:** Data tables + brief interpretation. Not an essay.

---

### `decision_log.md` — Decisions made and why

A running record of significant decisions. Each entry contains:
- The decision
- Context (what problem it solved)
- Rationale (why this option over others)
- Date
- Open questions it leaves

**Why it matters:** Prevents relitigating settled decisions and creates a paper trail for stakeholders.

---

### `status.md` — Current state

The one file to check if someone wants a quick read. Contains:
- Phase (Discovery / Design / Build / Shipped / Paused / Killed)
- Owner
- Jira ticket
- Groove Initiative + DoD
- Current blockers
- What we did last, what we're doing next
- Snow Site URL (if a bet brief has been deployed)

**Length:** Under 1 page. This is a snapshot, not a summary.

---

### `prd.md` — Product requirements

Detailed requirements for engineering. Generated by `/prd` from the other artifacts, or written manually. Contains:
- Problem statement and target users
- Success metrics (P0 must-haves vs. P1 nice-to-haves)
- Functional requirements (with priority tiers)
- Non-functional requirements (performance, security, accessibility)
- Out of scope
- Open questions

**Length:** Comprehensive. This is the one document where completeness beats brevity.

---

## Bet Lifecycle

```
02_parking_lot/    →    01_active_bets/    →    03_killed_ideas/
   (idea)                  (live bet)            (deliberate exit)
                               ↓
                          (shipped)
```

Moving a bet to `03_killed_ideas/` is not failure — it's recorded learning. The workspace treats explicit rejection as a first-class outcome.

---

## Starting a New Bet

Ask Claude to scaffold it:

> "Create a new bet called 'Revenue Reconciliation v2' in the reporting domain"

Claude will create the directory with empty template files. Then:
1. Run `/link-groove` to create a Groove Initiative and wire it to Jira
2. Fill in `hypothesis.md` and `problem_frame.md` first — before anything else
3. `evidence.md` and `decision_log.md` fill in as the bet progresses
