# Strategic Deck

Build a strategic HTML presentation in the style of Anil Somraj's Monetization Speed v3 deck. beat-by-beat HTML files, a Python assembler, scoped CSS, scroll-snap navigation, and a single deployable `snow-deploy/index.html`.

The style emphasises **visuals over text**, **plain human language**, and **compressed narrative** (12 main beats + small appendix). It has been battle-tested on FinE Leadership audiences and survived feedback that called out AI tone, em-dashes, and "X, not Y" rhetoric. This skill encodes those lessons.

## Trigger

Use when the user says any of:

- "build a strategic deck"
- "create a strategic HTML presentation"
- "write me a strategy review deck"
- "build a presentation in Anil's style"
- "/strategic-deck"
- explicit references to a strategic review with FinE-style audiences

Do NOT use this skill for:
- Pitch decks for individual bets. use `pitch-deck` instead
- Royalty domain pages. use `generate-royalty-page`
- One-off marketing pages or landing pages

## Outputs

A folder structure like:

```
{deck-folder}/
├── _assemble.py                   ← Python assembler (bundled with this skill)
├── beat-1-{name}.html             ← One file per slide
├── beat-2-{name}.html
├── ...
├── beat-N-{name}.html
├── snow-deploy/index.html         ← Combined output (regenerated by assembler)
└── README.md                      ← Deck overview, beat structure, deploy instructions
```

Plus optional: `feedback-overview.md`, `feedback-per-beat.md` for design rationale.

## Flow

### Step 1: Gather context

Use AskUserQuestion (one at a time) if these aren't already given:

**1a. Topic / framing.** "What's the deck about? One sentence. what's the strategic question or argument?"

**1b. Audience.** "Who's it for? Pick one: FinE Leadership / cross-org leadership / engineering peers / external customer / mixed exec audience / other."

**1c. Source material.** "What docs / Jira / Groove / pages should I read for context?" (paths or URLs). Read them with the Read tool.

**1d. Output folder.** "Where should the deck live?" Default: `domains/{area}/{topic}/strategy-review/` or wherever the source material is.

**1e. Snow slug.** "What Snow slug should this deploy to?" Default: `{topic}-review`.

### Step 2: Propose the beat outline

Read all source material. Then draft a beat outline of **8–12 main beats + 0–2 appendix beats**. The shape that has worked is:

| Beat | What it lands |
|------|---------------|
| 1 | Why this matters now (one provocation, one visual) |
| 2 | The tension / problem you exist to resolve |
| 3 | Where you sit (scope visual. phase strip, system map, or 5-circle centrality) |
| 4 | Strategy / OKRs / pillars (3 metric-led objectives is a strong default) |
| 5–N | Initiatives per objective (one beat per objective with KRs as panels) |
| N+1 | Proof point (live data, dashboard, or current-state evidence) |
| N+2 | Open questions (3 max. what you don't yet know) |
| Last | Asks (3 max. concrete, with owner and decision timeline) |

Adjust based on the topic. Resist going past 12 main beats; if it feels long, push detail into an appendix beat or drop it.

Present the outline. Use AskUserQuestion to get sign-off before writing any HTML.

### Step 3: Write each beat

Each beat is a **standalone HTML file** with its own `<style>`, a single `<section>`, and any visual it needs. The assembler scopes the CSS per beat so they don't collide.

**Use the bundled `template-beat.html`** as the starting scaffold. It contains:

- Font imports (Poppins + Lato)
- Global resets and body styling
- Default `section` style (scroll-snap, full viewport, centered, radial gradient)
- Common typography (`h2`, `.subtitle`, `.section-label`, `.inner`)

For each beat, copy the template, replace the title comment + section content, and add only the component CSS that beat needs. **Don't** carry every component's CSS into every beat. keep beats lean.

**Section anatomy** (use this skeleton inside `<section>`):

```html
<div class="inner">
  <div class="section-label">Short framing (small caps)</div>
  <h2>The headline</h2>
  <p class="subtitle">One-line context that a 12-year-old could parse.</p>
  <!-- Your visual / content here -->
</div>
```

Drop `section-label` if the headline already carries the framing.

### Step 4: Apply the visual library

Pick patterns from the library below. Each one has battle-tested CSS. Mix and match. don't reinvent.

#### Status pills + icons

```html
<span class="status-icon done">✓</span>     <!-- live / done -->
<span class="status-icon flight">▶</span>   <!-- in flight -->
<span class="status-icon planned">▸</span>  <!-- planned / next -->
<span class="status-icon cross">↪</span>    <!-- cross-functional input -->
```

Colors: `done #6ee7b7`, `flight #60a5fa`, `planned #fbbf24`, `cross #d4a8ff`.

#### KR / lever pills

```html
<span class="kr-pill">O1.KR3</span>
<span class="lever-pill">Configurable systems</span>
```

#### Objective panel (used for OKR slides)

```html
<div class="obj-panel" style="--obj-accent:#6ee7b7;">
  <div class="obj-head">
    <div class="obj-num">O1</div>
    <div class="obj-name">Reduce FinE TTM (60d → 21d)</div>
    <div class="obj-count">12 initiatives · 4 KRs</div>
  </div>
  <!-- KR rows or table here -->
</div>
```

Border-left 3px in the accent colour. Subtle bg `rgba(255,255,255,0.025)`. Radius 10px. Padding ~16px.

#### KR card stack (used for "initiatives per objective" slides)

```html
<div class="kr-stack">
  <div class="kr-panel" style="--kr-accent:#6ee7b7;">
    <div class="kr-header">
      <div class="kr-num">KR1</div>
      <div class="kr-lever-pill">Measurement</div>
      <div class="kr-desc">FinE TTM dashboard operational for all monetization solutions, reviewed in MS monthly cadence.</div>
    </div>
    <div class="initiatives">
      <div class="init">
        <span class="status done">✓</span>
        <span class="init-name">TTM Dashboard set-up</span>
        <span class="init-note">58d median, down 38%</span>
      </div>
    </div>
  </div>
</div>
```

#### Tension visual (two speeds + middle bridge)

Three columns side-by-side. Left and right have animated counter-rotating loops with a "fast" / "steady" label that stays upright. Middle node is the entity that bridges them. Used for any "we exist between two forces" framing.

#### Centrality diagram (1 of N circles)

Large central circle (the entity at the centre, glowing). 4 smaller circles overlap from N/E/S/W. Use when showing "X is one of N peer functions but central to the strategy."

#### Phase strip with brackets

A horizontal strip of N phase chips. Top bracket spans all phases for the end-to-end metric. Bottom brackets span sub-ranges for the metrics each owner cares about. Optional dashed feedback arc above. Used for any "where in the pipeline does this apply" framing.

#### Asks card (3 max)

```html
<div class="ask-grid">
  <div class="ask-card">
    <div class="ask-num">01</div>
    <div class="ask-headline">Centralise TTM measurement under FinE Ops</div>
    <div class="ask-divider"></div>
    <div class="ask-body">Mandate to instrument end-to-end. Unblocks the right-slice-of-TTM question.</div>
    <div class="ask-foot"><strong>FinE Leadership</strong><br>decision this cycle</div>
  </div>
  <!-- 2 more -->
</div>
```

Top accent bar via `::before` (4px gradient). Light tinted bg. Three columns side-by-side.

#### Open questions card (3 max)

Same shape as asks, but warmer accent (amber `#fbbf24`) and a "What this would change" callout.

### Step 5: Run the assembler

The bundled `_assemble.py` has a `BEATS` list:

```python
BEATS = [
    ("beat-1-name.html",  "beat-1",  "Slide title for nav"),
    ("beat-2-name.html",  "beat-2",  "Slide title for nav"),
    # ...
]
```

Edit the list to match your beats in order, then run:

```bash
cd {deck-folder}
python3 _assemble.py
```

Output: `snow-deploy/index.html`. Single file, ~100–150KB. Includes:

- Combined font imports
- Global resets (deduped)
- Per-beat scoped CSS
- All section markup
- Progress bar + nav dots + scroll script

### Step 6: Deploy to Snow

```bash
cd {deck-folder}
snow deploy -n "{snow-slug}" -b "./snow-deploy" --no-build -y
```

Returns `https://snow.spotify.net/s/{snow-slug}`.

Re-run after each edit. Snow caches by file hash so it's quick.

### Step 7: Write the README

Create `{deck-folder}/README.md` with:

- Audience
- Why this version exists (one paragraph)
- Beat table: # · Beat name · file · what it lands
- Local preview command
- Snow deploy command
- Re-assembly command after edits
- Linked artifacts (PRDs, OKRs, source docs)

Mirror the structure of `domains/alliance-strategy/monetization-speed/execution-review-fine-leadership/v3/README.md` if available.

## Design system reference

### Colours

| Role | Hex | Usage |
|------|-----|-------|
| Background | `#07162C` | Body bg |
| Body text | `#e8e8f0` | Default |
| Subtitle / muted | `#8899bb` | Subtitles, footers, italic notes |
| Done / live | `#6ee7b7` | O1 accent, status icons, "live" pills |
| In flight | `#60a5fa` | O2 accent, status icons, "in flight" pills |
| Planned | `#fbbf24` (or `#facc15`) | O3 accent, "next up" pills |
| Cross-functional | `#d4a8ff` | Cross-circle origin tags |
| Gap / risk | `#f87171` (or `#fca5a5`) | Owner gaps, "we don't have this" callouts |
| White | `#fff` | Headlines, key emphasis |

Don't introduce additional accent colours without a reason. Three accents per slide is the ceiling.

### Typography

```css
@import url('https://fonts.googleapis.com/css2?family=Poppins:wght@300;400;500;600;700;800;900&display=swap');
@import url('https://fonts.googleapis.com/css2?family=Lato:wght@300;400;700;900&display=swap');

body { font-family: 'Lato', -apple-system, sans-serif; }
h1, h2, .obj-name, .ask-headline { font-family: 'Poppins', sans-serif; }
```

| Element | Font | Size | Weight |
|---------|------|------|--------|
| Headline (`h2`) | Poppins | 32–36px | 800 |
| Subtitle | Lato | 15–17px | 400 |
| Section label | Lato | 12px | 700. uppercase, letter-spacing 0.18em |
| Body | Lato | 13–14px | 400 |
| Pill / tag | Lato | 9.5–11px | 700–800. uppercase, letter-spacing 0.06–0.14em |
| Hero | Poppins | 56–72px | 800–900 |

## Content rules. read this before writing copy

These are the hard-won lessons from real review feedback. Break them at your peril.

### What to avoid

- ❌ **Em-dashes (—).** Anil flagged em-dashes as the #1 AI-tone tell. Use a hyphen, comma, period, or colon instead. The assembler doesn't filter for you.
- ❌ **"X, not Y" inverse rhetoric.** *"Continuous controls, not hand-waved. Time cost is real, not optional."* Reads inhuman. Use direct phrasing.
- ❌ **Filler phrases.** *"In today's AI-native landscape..."* Cut.
- ❌ **Abstract phrasings without concrete examples.** *"Engineering effort scales with novel patterns, not novel offers."* Replace with: *"Build the shape once (sub, ad, royalty); every next instance is a config change."*
- ❌ **Status framing as questions.** *"Are we keeping pace?"* is fine as a Phase-2 callout. *"Is this strategy correct?"* on a strategy slide is not. answer the question.
- ❌ **Walls of bullet points.** If a slide has more than 5 bullets, find a visual or split it.
- ❌ **Hidden asks.** Asks belong on the slide that needs the decision, not buried in speaker notes.

### What to do instead

- ✓ **Concrete numbers and names.** *"60d → 21d."* *"Anil owns the dashboard."* *"Modular Subs WS5 ships Sept 2026."*
- ✓ **One message per slide.** If you can't summarise the slide in one sentence, split it.
- ✓ **Visual > paragraph.** Phase strip, panel, pill, card, diagram. find one.
- ✓ **Plain human language.** Read every line out loud. If you wouldn't say it to a colleague over coffee, rewrite it.
- ✓ **Three asks. Three open questions. Twelve slides.** These are the constraints that force compression.
- ✓ **Cross-functional credit.** When work comes from another team / circle, mark it with the cross icon (↪) and the origin name. Don't claim it as yours.

## Common failures & fixes

**The deck looks plain after assembly.** You forgot to scope a class. The assembler prefixes selectors with `section.beat-N`. If you wrote `section { ... }` it gets scoped correctly. If you wrote `body { ... }` it gets stripped (intentional). Use class selectors.

**Two beats fight over the same class.** Both used `.card`. Either rename to `.card-{beat-name}` or accept that the assembler will scope them to their own beat. they won't actually collide as long as the markup is inside the right `<section>`.

**Snow says "no changes."** Re-run `_assemble.py`. Snow uploads what's in `snow-deploy/index.html`, not your beat files.

**Em-dashes leaked back in.** Run `grep -c '—' beat-*.html` after writing each beat. Replace with a hyphen, comma, or period.

**The deck is 18 slides.** Push detail into an appendix beat or drop it. 12 main is the ceiling; 8–10 is healthy.
