## Prerequisites

Before running this command, verify the following are available:

1. **gcloud**: Run `which gcloud`. If missing, install via `brew install --cask google-cloud-sdk`
2. **curl**: Run `which curl`. If missing, install via `brew install curl`
3. **Jira credentials**: Ensure `JIRA_EMAIL` and `JIRA_API_TOKEN` are set in `.env.local` at the workspace root. If missing, create the file with your Jira email and an API token from https://id.atlassian.com/manage-profile/security/api-tokens
4. **gcloud auth (calendar scopes)** (for `--kickoff`): Run `gcloud auth application-default print-access-token`. If missing or expired, run:
   ```
   gcloud auth application-default login \
     --scopes=https://www.googleapis.com/auth/calendar,https://www.googleapis.com/auth/cloud-platform
   ```

If any prerequisite is missing, walk the user through setting it up before proceeding.

# Engineering Handoff Skill

Transform a bet into actionable Jira work items and schedule a kickoff meeting with the engineering team.

## Overview

This skill reads bet documents (problem_frame.md, prd.md, hypothesis.md) and:
1. Detects **prototypes** (from `/rapidly`) and incorporates them into epic descriptions and stories
2. Creates one or more Jira **Epics** based on milestones/phases from the PRD
3. Optionally breaks down each Epic into **User Stories** with acceptance criteria — enriched with prototype flows and screens when available
4. Optionally books a **kickoff meeting** with the engineering team (includes prototype walkthrough if available)
5. Updates the bet's `status.md` with links to created tickets

**Presentations:** If the bet has a `presentation.html` or presentation markdown file stored alongside the bet directory, you can export it to Google Slides (`/export-slides`) or PowerPoint (`/export-pptx`) for use in the kickoff meeting.

## Configuration

- **Environment Variables:** `JIRA_EMAIL`, `JIRA_API_TOKEN` from `./.env.local`
- **Meeting Booking:** Uses gcloud ADC for Calendar API

## Instructions

### Step 1: Parse Arguments

```
/handoff domains/booking/01_active_bets/automated reconciliation
/handoff domains/booking/01_active_bets/automated reconciliation --project MYPROJ
/handoff domains/booking/01_active_bets/automated reconciliation --stories
/handoff domains/booking/01_active_bets/automated reconciliation --kickoff "Nate, anna@spotify.com"
/handoff domains/booking/01_active_bets/automated reconciliation --project MYPROJ --stories --kickoff "Team Lead"
```

**Arguments:**
- `<bet-path>` - Path to the bet directory (required)
- `--project KEY` - Jira project key (e.g. `FTI`, `TRUST`, `PAY`). If omitted, check `status.md` for an existing project key, then ask the user.
- `--stories` - Generate user story breakdown for each epic
- `--kickoff "attendees"` - Book a kickoff meeting with specified attendees
- `--dry-run` - Preview what would be created without actually creating

**Resolving the project key:**

If `--project` is not provided:
1. Check `status.md` for any existing Jira ticket keys (e.g. `ABC-123`) and extract the project prefix
2. If found, confirm with the user: *"I found existing tickets in project ABC — use that for new epics?"*
3. If not found, use `AskUserQuestion` to ask: *"Which Jira project should I create the epics in?"*

If no path provided, prompt the user to select from available bets.

### Step 2: Read Bet Documents

Load the core bet documents:

```bash
# Find bet documents
ls "${BET_PATH}/"*.md
```

Read these files in order of importance:
1. `prd.md` - Primary source for epics (milestones, scope, success metrics)
2. `problem_frame.md` - Context and constraints
3. `hypothesis.md` - Success criteria and disproof conditions
4. `status.md` - Current state and existing Jira tickets

**Check for prototypes:**

Look for a `prototype/` subdirectory in the bet path (generated by the `/rapidly` skill):

```bash
ls "${BET_PATH}/prototype/" 2>/dev/null
```

If the directory exists, read the available prototype artifacts:
- `prototype/cpm.json` — Canonical Prototype Model (structured intent, jobs, flows)
- `prototype/prototype_plan.json` — Screen/flow plan with screen list
- `prototype/figma_make_prompt.txt` — Figma Make prompt
- `prototype/index.html` — Interactive HTML prototype

If `cpm.json` exists, read it and extract:
- **Initiative type** (`classification.initiative_type`) — informs epic scoping
- **Jobs to be done** (`jobs`) — map to user stories
- **Flows** (`flows`) — map to implementation tasks
- **Prototype type** (`prototype.strategy.recommended_type`) — referenced in epic descriptions
- **Open questions** (`review.open_questions`) — surface as risks/blockers

If `index.html` exists, note its path for linking in epic descriptions and kickoff materials.

**Extract key information:**

| Source | Extract |
|--------|---------|
| PRD - Milestones | → Epics (one per phase/milestone) |
| PRD - Goals | → Epic acceptance criteria |
| PRD - Constraints | → Technical notes on tickets |
| PRD - Risks | → Risk callouts in descriptions |
| Problem Frame | → Epic descriptions |
| Hypothesis | → Definition of Done criteria |
| Status | → Existing Jira ticket to link as parent |
| Prototype - CPM | → Jobs and flows inform story breakdown |
| Prototype - Flows | → Implementation tasks and screen inventory |
| Prototype - HTML | → Link in Design & Build epic descriptions |
| Prototype - Open Questions | → Risk callouts and blockers |

### Step 3: Generate Epic Structure

From the PRD milestones, create an Epic for each phase. Typical bet phases:

| Phase | Epic Title Pattern |
|-------|-------------------|
| Discovery | `[Bet Name]: Discovery & Baseline` |
| Design | `[Bet Name]: Design & Architecture` |
| Build | `[Bet Name]: Implementation` |
| Parallel Run | `[Bet Name]: Validation & Parallel Run` |
| Rollout | `[Bet Name]: Production Rollout` |

**Epic structure:**

```
Summary: [Bet Name]: [Phase Name]

Description:
## Overview
[Extract from PRD phase description]

## Goals
[Goals from PRD relevant to this phase]

## Exit Criteria
[From PRD milestone exit criteria]

## Constraints
[Relevant constraints from PRD]

## Prototype
[If prototype exists — include for Design, Build, and Validation epics]
- Interactive prototype: [path or link to prototype/index.html]
- Prototype type: [clickable_ui / data_flow_diagram / architecture_diagram / hybrid_bundle]
- Screens/flows covered: [list from prototype_plan.json]
- Figma Make prompt: [path to prototype/figma_make_prompt.txt]

## Related Documents
- Problem Frame: [link to Snow site or markdown]
- PRD: [link]
- Hypothesis: [link]
```

**Prototype integration per phase:**

If a `prototype/` directory was found in Step 2, enrich the relevant epics:

| Phase | Prototype Integration |
|-------|----------------------|
| Discovery | Reference CPM open questions as discovery tasks. Link CPM jobs to validate. |
| Design | Include full prototype section. Link to `index.html` as design reference. Reference Figma Make prompt for high-fidelity design work. List screens from `prototype_plan.json` that need design review. |
| Build | Include prototype section. Map CPM flows to implementation tasks. Reference prototype screens as acceptance visual — "implementation should match prototype behavior". |
| Validation | Reference prototype as baseline for UAT. Include CPM success metrics as validation criteria. |
| Rollout | No prototype section needed. |

### Step 5: Create Jira Epics

Load credentials and create each epic:

```bash
source ./.env.local

# Look up current user's account ID
JIRA_ACCOUNT_ID=$(curl -s -X GET \
  -H "Authorization: Basic $(echo -n "$JIRA_EMAIL:$JIRA_API_TOKEN" | base64)" \
  "https://spotify.atlassian.net/rest/api/3/user/search?query=$JIRA_EMAIL" \
  | jq -r '.[0].accountId')
```

**Create Epic:**

```bash
curl -s -X POST \
  -H "Authorization: Basic $(echo -n "$JIRA_EMAIL:$JIRA_API_TOKEN" | base64)" \
  -H "Content-Type: application/json" \
  -d '{
    "fields": {
      "project": {"key": "'"$PROJECT_KEY"'"},
      "summary": "[Bet Name]: [Phase Name]",
      "description": {
        "type": "doc",
        "version": 1,
        "content": [
          {
            "type": "paragraph",
            "content": [{"type": "text", "text": "[Description from PRD]"}]
          }
        ]
      },
      "issuetype": {"name": "Epic"},
      "assignee": {"accountId": "'"$JIRA_ACCOUNT_ID"'"}
    }
  }' \
  "https://spotify.atlassian.net/rest/api/3/issue"
```

Alternatively, use the MCP tool:

```
mcp__atlassian-mcp__create_ticket(
  project_key="$PROJECT_KEY",
  summary="[Bet Name]: [Phase Name]",
  description="[ADF formatted description]",
  issue_type="Epic",
  assignee="[email]"
)
```

**Capture the created epic keys** (e.g., ABC-542, ABC-543, etc.) for story creation.

### Step 6: Generate User Stories (if --stories)

For each Epic, generate user stories based on the phase activities and PRD details.

**Story generation patterns by phase:**

| Epic Phase | Story Patterns |
|------------|----------------|
| **Discovery** | "Baseline current [metric]", "Interview [stakeholder group]", "Document current process" |
| **Design** | "Design [component]", "Create architecture diagram", "Define API contracts" |
| **Build** | "Implement [feature]", "Add [integration]", "Create [automation]" |
| **Validation** | "Run parallel test with [scope]", "Validate accuracy for [case]", "Gather user feedback" |
| **Rollout** | "Deploy to [env]", "Monitor [metric]", "Create runbook" |

**Additional stories when prototype exists:**

If a `prototype/` directory was found in Step 2, generate additional stories derived from the prototype artifacts:

| Epic Phase | Prototype-Derived Stories |
|------------|--------------------------|
| **Discovery** | "Validate CPM jobs-to-be-done with stakeholders", "Review prototype open questions: [list from cpm.json]" |
| **Design** | "Review interactive prototype with design team" (link to `index.html`), "Refine Figma designs from Make prompt" (link to `figma_make_prompt.txt`), "Design review for each prototype screen: [screen names from prototype_plan.json]" |
| **Build** | One story per CPM flow: "Implement [flow name] ([N] steps)" with acceptance criteria derived from flow steps. Link prototype screen as visual reference for each story. |
| **Validation** | "UAT against interactive prototype" (compare implementation to `index.html`), "Validate CPM success metrics: [metrics from cpm.json]" |

When generating Build stories from CPM flows, use the flow's steps as acceptance criteria:

```
Summary: Implement [flow name]

Description:
## User Story
As a [flow actor], I want to [flow purpose] so that [linked job success_outcome].

## Prototype Reference
- Interactive prototype: [path to index.html]
- Relevant screens: [screen IDs from prototype_plan.json matching this flow]

## Acceptance Criteria (from prototype flow steps)
- [ ] [Step 1: actor performs action → expected result]
- [ ] [Step 2: actor performs action → expected result]
- [ ] [Step 3: actor performs action → expected result]

## Definition of Done
- [ ] Implementation matches prototype behavior
- [ ] Code reviewed and merged
- [ ] Tests passing
```

**Story structure:**

```
Summary: [Action verb] [specific deliverable]

Description:
## User Story
As a [role], I want to [action] so that [benefit].

## Acceptance Criteria
- [ ] [Specific, testable criterion 1]
- [ ] [Specific, testable criterion 2]
- [ ] [Specific, testable criterion 3]

## Technical Notes
[Any constraints or implementation hints from PRD]

## Definition of Done
- [ ] Code reviewed and merged
- [ ] Tests passing
- [ ] Documentation updated
```

**Create Story as subtask of Epic:**

```bash
curl -s -X POST \
  -H "Authorization: Basic $(echo -n "$JIRA_EMAIL:$JIRA_API_TOKEN" | base64)" \
  -H "Content-Type: application/json" \
  -d '{
    "fields": {
      "project": {"key": "'"$PROJECT_KEY"'"},
      "parent": {"key": "'"$EPIC_KEY"'"},
      "summary": "[Story title]",
      "description": {<ADF document>},
      "issuetype": {"name": "Story"}
    }
  }' \
  "https://spotify.atlassian.net/rest/api/3/issue"
```

Or use MCP:

```
mcp__atlassian-mcp__create_ticket(
  project_key="$PROJECT_KEY",
  summary="[Story title]",
  description="[Description]",
  issue_type="Story",
  parent_issue_key="$EPIC_KEY"
)
```

## Tips

- **Run `/rapidly` before `/handoff`** to generate a prototype — the handoff will automatically detect it and produce richer epics with prototype-linked stories
- **Start with `--dry-run`** to preview the epic structure before creating
- **Use `--stories` for mature bets** with well-defined PRD milestones — especially powerful when a prototype exists, as CPM flows become implementation stories with visual acceptance criteria
- **Include EM and TL in `--kickoff`** for engineering alignment
- **Review generated stories** — they're starting points, not final specifications
- **Link to existing parent ticket** if the bet already has a Jira Story from scaffolding
